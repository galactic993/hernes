# 設計書読み取りスキル（画面設計書 → spec）

## 目的

人手で作られた Excel 画面設計書を、機械判定可能な仕様へ変換する。
**セル値のテキスト抽出ではなく、シートを画像として視覚的に読む。**
これにより、画面概要シートの埋め込みキャプチャ・I/Oフロー図・結合セル・色の意味（黄=未記入, 青=ヘッダ）を取りこぼさない。

## 入力

- 対象の画面設計書(.xlsx)。パスは `specs/<feature>/design-source.md` に明記
- `docs/constitution.md`

## 出力

- `design/rendered/<doc>/page-*.png`（レンダリング結果）
- `specs/<feature>/intent.yaml`（未作成なら下書き）
- `specs/<feature>/spec.md`

## レンダリング方針（必ず守る）

設計書は **「xlsx → PDF → PNG」と必ず PDF を中継** して画像化する。直接スクショやセル抽出はしない。

1. **PDFを挟む**理由: PDF段階で**大判化（1シート=1ページ）**でき、A4で細切れにならず、1シートを1枚の高解像度画像として通して読める。
2. **大判・高解像度で出す**: `scripts/render-design.sh` は既定で `SinglePageSheets`（1シート1ページ）で出力する。文字が小さければ `DPI=200`（既定150）に上げる。
3. **ソースの .xlsx を絶対に書き換えない**:
   - openpyxl 等でページサイズ（A2化など）を変えて保存し直すと、**埋め込みの I/Oフロー図・SmartArt・シェイプが消える**（検証で確認済み）。
   - 用紙の大判化は LibreOffice 側（`render-design.sh`）でのみ行い、元ファイルは読み取り専用として扱う。
   - cf. [docs/decision-log/ADR-0002-design-doc-as-image.md]

## 手順

1. `scripts/render-design.sh <xlsx>`（= `make design DESIGN=<xlsx>`）を実行し、各シートを大判 PNG 化する。
   - 文字が小さい場合は `DPI=200 scripts/render-design.sh <xlsx>` で再実行。
   - 1シートが巨大すぎて扱いにくい場合のみ `SINGLE_PAGE=0`（印刷設定どおりの分割）に切り替える。
2. ページ画像を**すべて視覚的に読む**。特に次のシートを重点的に：
   - **表紙 / 変更履歴** … 対象画面・版・変更経緯
   - **画面概要** … 画面の目的・利用権限一覧・キャプチャ画像（UIの実体）・I/Oフロー図（画面⇔サーバ⇔データ、参照テーブル）
   - **イベント記述書** … EVENT No と処理内容（権限判定・取得・検索・登録）
   - **項目記述書** … 項目名 / 種別 / EVENT No / 制御区分(JavaScript=フロント, PHP=サーバ) / 制御内容 / メッセージ
   - **参照仕様** … 抽出テーブル・抽出項目・抽出条件
   - **メッセージ一覧** … 条件 / 区分(エラー/サクセス) / メッセージ
3. 上記を `spec.md` の要件（FR / NFR / SEC）・受け入れ条件の種・画面項目・メッセージへ写経する。
4. 制御内容（必須・桁数・有効値・形式）は必ず **項目No・EVENT No・出典シート名** を併記し、後工程で辿れるようにする。
5. 画像から判断できない点は推測せず、`questions.md` / spec の「未解決」へ回す。

## 品質基準

- すべての画面項目・メッセージ・イベントが spec に現れる
- 各要件が「どのシートの何行 / どの項目No」由来か辿れる
- キャプチャ画像のUI要素が項目記述書と矛盾しない
- I/Oフロー図がレンダリング結果に**写っている**ことを確認する（写っていなければソースを書き換えてしまった疑い → 元ファイルから再レンダリング）

## 注意

- 元設計の実装スタック表記（Laravel / PHP）は**参考**。本リポジトリでは Hono / React / TypeScript で作り直す。
- 制御区分 JavaScript はフロント(React)、PHP はサーバ(Hono) のバリデーションに対応づける。両者は `@hernes/shared` の同一スキーマを使う。
