# プロジェクト憲法

> 全自動ループにおいて「AIが勝手に変えてはいけない価値判断」。
> 仕様生成・計画・実装・レビューのすべてに効く。最初に確定させる。

> **機械可読版**: この散文は [`governance/constitution.yaml`](../governance/constitution.yaml) に
> ミラーされ、立法（`governance/rules/`）・司法（`governance/checks/`）・行政（`make verify`）すべてが準拠する。
> 各原則は「書かれている」だけでなく「効いている」（違反したら `make govern` が CI を落とす）状態を目指す。
> 統治モデルの全体は [docs/governance.md](governance.md) / [ADR-0005](decision-log/ADR-0005-governance-three-powers.md)。
> 改正は**人間のみ**可（AI は提案まで）。

## プロダクト原則

1. **ユーザー価値が最優先**
   - すべての機能はユーザーの課題に対応づく。社内都合だけでは不十分。

2. **テストファースト**
   - 新しい挙動は実装より前にテストを書く。
   - バグ修正にはリグレッションテストを付ける。

3. **小さな変更**
   - 大きな書き直しより、小さくレビュー可能な変更を好む。
   - 各タスクは独立して merge できる粒度にする。

4. **観測可能な挙動**
   - 重要なフローはログ・メトリクス・トレースを持つ。
   - 失敗は診断可能であること。

5. **セキュリティとプライバシー**
   - ソースに秘密情報を置かない。
   - 機微データをログに出さない（個人情報の非ログ出力）。
   - 認可チェックはテストで担保する。

6. **設計書・仕様が真実の源**
   - 人手の Excel 設計書（画面設計書・テーブル定義書）が要件の起点。
   - 実装と仕様が食い違ったら、どちらかを**意図的に**更新する。コードを黙って乖離させない。
   - メッセージ・列挙値・バリデーション規則は `@hernes/shared` に単一定義する（直書き禁止）。

## 品質ゲート

機能は次を満たすまで「完了」にできない：

- 受け入れテストが存在する
- コアロジックを unit / integration テストが覆う
- `make verify`（lint + typecheck + test）が通る
- リスクと限界が文書化されている
- 本番変更にはロールバック/復旧経路がある
